# Supermarket Ops Agent

> Run an entire Indian kirana store from a Telegram chat — with an agent, not a menu.

A conversational agent that operates a small Indian grocery/kirana store end-to-end.
The owner runs the whole shop in plain language over Telegram: receiving stock,
cutting GST-correct bills, running customer credit (_khata_), closing the day, and
generating PDF invoices and PowerPoint analysis decks on demand.

There is no web app, no admin panel, no forms. **The chat is the product.** The model
reasons over messy, terse shopkeeper requests and orchestrates a set of thin, typed
tools; every business rule is enforced in the tool/data layer, never hoped for in a
prompt.

**Telegram bot:** [@SupermarketOpsAgentBot](https://t.me/SupermarketOpsAgentBot)

## Highlights

- **Agent-first, not a router.** Natural language goes through the model plus a set of
  authored tools. There is no `if/elif` intent router — the model decides which tools
  to call and chains several within a single turn.
- **Grounded.** Prices, GST slabs and stock always come from the database via tools.
  The agent never invents a product, price or quantity.
- **GST-correct.** MRP is treated as GST-inclusive (Indian retail convention); the
  taxable value is backed out, CGST/SGST are split, and everything is computed in
  integer paise so nothing is ever lost to rounding.
- **Safe by construction.** Oversell guard, idempotent billing, concurrency safety and
  khata rules are enforced inside database transactions — the model _cannot_ bypass them.
- **Real artifacts.** GST invoices as clean PDFs and analysis decks as PPTX files with
  native charts, generated by the agent's own tools and delivered as Telegram documents.
- **Durable memory.** Standing preferences and shop identity persist across chats and
  restarts — they live in the database, outside the conversation window.
- **Speaks the owner's language.** Understands and replies in English, Hindi, Tamil, or
  Hinglish, matching whatever the owner types.
- **Reads product photos.** Snap a picture of a product and the agent identifies it
  (Claude vision), matches it to the catalogue, and acts on it.
- **Works proactively.** Scheduled khata payment reminders and a weekly analysis deck
  are pushed to the owner automatically.

## Architecture

```
Telegram (grammY) ──── text · photo (Claude vision) ──┐        ┌── scheduled jobs (cron)
      │  message  ▲  reply + documents                │        │   khata reminders · weekly deck
      ▼           │                                   ▼        ▼
  Agent loop (Claude Agent SDK · one query() per turn, resumed by session id)
      │  observe → reason → act → feed result back → continue
      ▼
  In-process tool surface (24 tools across skill modules)
      │  every business rule enforced here
      ▼
  SQLite (WAL) — durable store: products, bills, khata, preferences, shop
```

### Harness — and why

**[Claude Agent SDK for TypeScript](https://code.claude.com/docs/en/agent-sdk)**
(`@anthropic-ai/claude-agent-sdk`), driving **Claude Opus 4.8**.

Why this harness:

- **It runs the observe → reason → act loop for us.** `query()` gives a real agentic
  loop — the model calls a tool, sees the result, reasons, and calls the next tool,
  chaining as many as a turn needs. We don't hand-roll the loop or a state machine.
- **First-class custom tools.** Tools are defined in-process with `tool()` + Zod schemas
  and served via an in-process MCP server (`createSdkMcpServer`). No subprocess, no
  network hop — tool handlers run in the same process and talk directly to SQLite.
- **Locked-down permissions.** `permissionMode: 'dontAsk'` plus an explicit
  `allowedTools` list means the model can reach **only** the store's tools — never the
  host shell or filesystem.
- **Session continuity for free.** Each turn resumes the prior conversation by session
  id, so multi-turn bills and follow-ups carry across messages without us persisting
  chat history ourselves.

This is deliberately _not_ a LangGraph-style node-per-command graph. The whole point of
the task is to let the model orchestrate a well-designed tool surface; a graph would
re-introduce the hardcoded routing the brief explicitly warns against.

### The control loop

One `query()` per user turn (see [`src/agent/respond.ts`](src/agent/respond.ts)):

```ts
for await (const message of query({
  prompt: text,
  options: {
    model: config.model,
    systemPrompt: `${SYSTEM_PROMPT}\n\n${buildOwnerContext()}`, // durable memory injected here
    mcpServers: { store: server }, // in-process tools bound to this chat
    allowedTools: toolNames, // only the store tools
    permissionMode: "dontAsk", // deny anything not pre-approved
    resume, // continue this chat's conversation
    maxTurns: 20,
  },
})) {
  /* collect final text + any generated files */
}
```

- **Streaming input mode doesn't fit an event-driven bot**, so instead of holding a
  generator open we resume the session on each incoming Telegram message. Conversation
  state lives in the SDK session; durable _store_ state lives in SQLite.
- **Chat context is injected, not trusted to the model.** The tool surface is rebuilt
  per turn by a factory (`buildStoreServer({ chatId, attachments })`) so chat-scoped
  tools (billing) close over the Telegram chat id. The model can never spoof _whose_
  bill or khata it is acting on.
- **Files flow back out-of-band.** Document tools push generated PDFs/PPTX into a
  per-turn outbox; the bot sends the text reply, then each artifact as a document.

### Skill / tool design

Tools are grouped by skill/domain module and composed into a single in-process MCP
server ([`src/tools/store-server.ts`](src/tools/store-server.ts)). Each tool is thin and
typed; the _composition_ is left to the model.

| Skill module                                                   | Tools                                                                                                | Responsibility                                               |
| -------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| **Inventory** ([`inventory.ts`](src/tools/inventory.ts))       | `get_stock`, `add_product`, `receive_stock`, `list_low_stock`, `reorder_suggestions`                 | Catalogue, stock-in, reorder alerts + velocity-based restock |
| **Billing** ([`billing.ts`](src/tools/billing.ts))             | `add_line_item`, `update_line_item`, `remove_line_item`, `view_bill`, `finalize_bill`, `cancel_bill` | Multi-turn draft bills, GST, finalize                        |
| **Khata** ([`khata.ts`](src/tools/khata.ts))                   | `khata_charge`, `khata_payment`, `khata_balance`, `khata_outstanding`, `khata_reminders`             | Customer credit ledger + follow-ups                          |
| **Settings** ([`settings.ts`](src/tools/settings.ts))          | `set_shop_details`, `get_shop_details`                                                               | Shop identity for invoices                                   |
| **Documents** ([`documents.ts`](src/tools/documents.ts))       | `generate_invoice_pdf`, `generate_analysis_deck`                                                     | Real PDF / PPTX artifacts                                    |
| **Preferences** ([`preferences.ts`](src/tools/preferences.ts)) | `set_preference`, `get_preferences`, `forget_preference`                                             | Standing owner preferences (durable memory)                  |
| **Reports** ([`reports.ts`](src/tools/reports.ts))             | `sales_report`                                                                                       | Daily close / period sales summary                           |

Design principles:

- **Thin tools, smart model.** A tool does one small, well-typed thing (`add_line_item`
  adds one line). The model composes them — e.g. "make a bill: 2kg sugar, 1 atta,
  4 Maggi, UPI" becomes several `add_line_item` calls then `finalize_bill`.
- **Business rules live in the tool/repo layer.** The repositories
  ([`src/db/*.ts`](src/db)) own the invariants and run inside `db.transaction(...)`.
  Tools translate model requests into repo calls and relay refusals.
- **Refusals are first-class.** A `StoreError` thrown anywhere in the tool/repo layer
  (oversell, below-cost, unknown khata, invalid GST slab, …) is caught by a small
  `guard()` wrapper and returned to the owner as a clear message — the guardrail is
  _enforced_, the model just narrates it.
- **Clarification comes from the model.** When a keyword is genuinely ambiguous
  (`atta` → Aashirvaad 5kg _or_ loose atta), the repo returns the matches and the model
  asks which one — there is no hardcoded disambiguation branch.

### Data model

SQLite (WAL mode), money stored as **integer paise** throughout so GST maths stays
exact. Schema in [`src/db/schema.ts`](src/db/schema.ts):

| Table                              | Purpose                                                                 |
| ---------------------------------- | ----------------------------------------------------------------------- |
| `products`                         | SKU, unit, cost, MRP (GST-inclusive), GST rate, HSN, qty, reorder level |
| `bills` / `bill_items`             | Draft → final bills with per-line GST snapshots                         |
| `khata_customers` / `khata_ledger` | Credit customers; balance is the derived sum of ledger deltas           |
| `preferences`                      | Standing owner preferences (key → value)                                |
| `shop_config`                      | Shop name / GSTIN / address for invoices                                |
| `processed_updates`                | Telegram `update_id` de-duplication (idempotency)                       |

## How each hard part is solved

The brief calls out nine "hard parts" that separate a real agent from a toy. Each is
handled in code, not in the prompt:

| #   | Hard part                  | How it's solved                                                                                                                                                                                         | Where                                                                                            |
| --- | -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| 1   | **Grounding**              | Every price/stock/GST fact is read from SQLite via tools; the system prompt forbids inventing them and the tools are the only source of truth.                                                          | [`inventory.ts`](src/tools/inventory.ts), [`system-prompt.ts`](src/agent/system-prompt.ts)       |
| 2   | **Oversell guard**         | Stock is decremented with a conditional `UPDATE … WHERE qty >= needed`; if it changes 0 rows the transaction throws and rolls back. Enforced at the tool layer, refused with a clear message.           | [`bills.ts` → `finalizeDraft`](src/db/bills.ts)                                                  |
| 3   | **GST correctness**        | MRP is GST-inclusive → taxable value backed out, tax split into CGST/SGST, exact integer-paise rounding, legible per-slab breakup on the bill and invoice.                                              | [`domain/gst.ts`](src/domain/gst.ts)                                                             |
| 4   | **Multi-turn bills**       | A draft persists in the DB across messages; items merge, update and remove; **stock is decremented only on finalize**.                                                                                  | [`bills.ts`](src/db/bills.ts)                                                                    |
| 5   | **Idempotency**            | Two layers: every Telegram `update_id` is recorded once (`processed_updates`), and finalize consumes the draft (draft → final), so a retried finalize finds no open draft and can't double-bill.        | [`bot.ts`](src/telegram/bot.ts), [`bills.ts`](src/db/bills.ts)                                   |
| 6   | **Concurrency**            | WAL + `BEGIN IMMEDIATE` transactions and the atomic conditional decrement mean two bills — or a sale plus a stock-in — can't corrupt stock.                                                             | [`db/index.ts`](src/db/index.ts), [`bills.ts`](src/db/bills.ts)                                  |
| 7   | **Guardrails**             | Below-cost sales, invalid GST slabs, duplicate SKUs, and settling a non-existent khata are all refused in the repo layer. There is no destructive "delete stock" tool.                                  | [`products.ts`](src/db/products.ts), [`khata.ts`](src/db/khata.ts)                               |
| 8   | **Real artifacts**         | GST invoices rendered with pdfmake (HSN, CGST/SGST, amount in words); analysis decks built with pptxgenjs as native PowerPoint charts — both generated by tools and sent as Telegram documents.         | [`invoice-pdf.ts`](src/domain/invoice-pdf.ts), [`analysis-deck.ts`](src/domain/analysis-deck.ts) |
| 9   | **Memory across sessions** | Standing preferences and shop identity live in the database and are injected into the system prompt on every turn (rebuilt from the DB), so they apply in a brand-new `/new` chat and survive restarts. | [`owner-context.ts`](src/agent/owner-context.ts), [`preferences.ts`](src/db/preferences.ts)      |

## Beyond the brief — stretch features

Six of the optional stretch goals are implemented, most reusing infrastructure already
in place:

| Feature                         | What it does                                                                                                                                                  | Where                                                                           |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **Multi-language**              | Understands and replies in English / Hindi / Tamil / Hinglish, matching the owner's current message per turn.                                                 | [`system-prompt.ts`](src/agent/system-prompt.ts)                                |
| **Reorder from sales velocity** | Estimates each item's daily sales rate, days of cover left, and a suggested order quantity — smarter than a fixed reorder level.                              | [`analytics.ts`](src/db/analytics.ts), [`inventory.ts`](src/tools/inventory.ts) |
| **Branded invoice PDF**         | Coloured header band, auto-generated monogram, tagline, accent styling — on top of the GST-correct layout.                                                    | [`invoice-pdf.ts`](src/domain/invoice-pdf.ts)                                   |
| **Photo → identify item**       | Send a product photo; Claude vision identifies it, the result flows into the normal agent pipeline (grounded via `get_stock`) and can be billed or restocked. | [`vision.ts`](src/agent/vision.ts), [`bot.ts`](src/telegram/bot.ts)             |
| **Khata payment reminders**     | On demand ("who should I remind?") the agent lists debtors with debt age and drafts polite messages; a daily cron also nudges the owner proactively.          | [`reminders.ts`](src/domain/reminders.ts), [`scheduler.ts`](src/scheduler.ts)   |
| **Scheduled weekly deck**       | A weekly cron generates the analysis deck and auto-sends it to the owner (skips quiet weeks).                                                                 | [`scheduler.ts`](src/scheduler.ts), [`deck-data.ts`](src/domain/deck-data.ts)   |

Design notes:

- **Vision runs outside the agent loop.** A photo is identified by a one-shot Claude
  vision call; the label is fed back in as text so grounding stays in the agent's hands
  (it still uses `get_stock` to match the catalogue).
- **Proactive jobs need the owner's chat id.** It's captured on interaction and stored
  as an _internal_ (`_`-prefixed) preference that is hidden from the model, or set via
  `OWNER_CHAT_ID`. Schedules and timezone are configurable (IST by default).

## Getting started

### Prerequisites

- Node.js 20+ (developed on 24)
- A Telegram bot token from [@BotFather](https://t.me/BotFather)
- An Anthropic API key

### Setup

```bash
git clone <repo-url>
cd Bigmantra_Hiring_Task
npm install

cp .env.example .env      # then fill in the two secrets
npm run seed              # seed 12 real kirana SKUs with correct HSN + GST
npm run dev               # start the bot (auto-reloads on changes)
```

`.env`:

```
ANTHROPIC_API_KEY=sk-ant-...
TELEGRAM_BOT_TOKEN=123456:ABC-your-bot-token

# optional
# MODEL=claude-opus-4-8            # main agent model
# VISION_MODEL=claude-haiku-4-5    # model used to identify products from photos
# DB_PATH=./data/store.db
# OWNER_CHAT_ID=123456789          # chat for proactive messages (else captured on first use)
# TZ=Asia/Kolkata                  # timezone for scheduled jobs
# KHATA_REMINDER_CRON=0 10 * * *   # daily 10:00 khata reminder
# KHATA_REMINDER_MIN_DAYS=3        # only remind about debts at least this old
# WEEKLY_DECK_CRON=0 9 * * 1       # Monday 09:00 weekly deck
```

Then message your bot on Telegram and send `/start`.

## Testing

**Automated suite (Vitest).** 46 tests cover the parts where correctness is
non-negotiable — the same hard parts called out above:

| Area                   | What's asserted                                                        |
| ---------------------- | ---------------------------------------------------------------------- |
| GST maths              | inclusive back-out, `taxable + cgst + sgst == gross`, per-slab breakup |
| Bill lifecycle         | draft merge/edit/cancel, stock **not** reserved until finalize         |
| Oversell + rollback §4 | a short line throws and rolls the **whole** bill back atomically       |
| Idempotent finalize §5 | the draft is consumed, so a retry can't decrement stock twice          |
| Khata rules §7         | derived balance, name normalization, refuse-unknown-customer           |
| Analytics              | sales summary by mode/tax, top items, sales velocity & days-of-cover   |
| Product guardrails     | valid unit/GST, MRP ≥ cost, no duplicates, resolve/ambiguity           |

Each test file runs against its own isolated **in-memory SQLite** (`DB_PATH=:memory:`),
so the suite never touches real data and needs no API keys.

```bash
npm test            # 46 tests, ~1s
npm run coverage    # + coverage report
```

**Quality gates** (all clean):

```bash
npm run typecheck   # tsc --noEmit

```

## Design decisions & trade-offs

- **Money in integer paise.** Storing rupees as floats invites rounding drift in GST
  maths; integers make every total exact and reproducible.
- **Balance is derived, not stored.** A khata balance is the running sum of ledger
  deltas, so it can never disagree with its own history.
- **Per-turn tool factory over a global tool server.** Rebuilding the tool surface each
  turn (cheap) lets chat-scoped tools close over the Telegram chat id from Telegram —
  not from model-supplied arguments — which is both correct and a security boundary.
- **Draft bills live in the DB, not the session.** This is what makes multi-turn
  billing, "decrement only on finalize", and idempotent finalize fall out naturally.
- **Soft warning at add, hard refusal at finalize.** Overselling is flagged early for a
  good UX, but only _enforced_ atomically at finalize where the stock actually moves.
- **Deliberately no `delete_product` / `delete_stock` tool.** Honours the "don't delete
  stock" guardrail; corrections are made by receiving stock or adjusting a bill.
- **Internal state is hidden from the model.** App state such as the owner's chat id is
  stored as an `_`-prefixed preference and filtered out of the prompt, so it never leaks
  into or gets altered by the conversation.
- **Vision decoupled from the agent.** Photo identification is a separate one-shot call;
  its text result re-enters the normal loop, keeping grounding and tool use unchanged.
